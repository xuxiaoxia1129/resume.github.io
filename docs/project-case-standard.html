项目案例页展示标准 ​

本文档用于统一个人简历网站中所有项目案例页的写作、结构、视觉证据和表达边界。后续无论是澜思、利多星、个人项目、AI 项目还是知识库项目，都应按此标准组织，确保页面面向 HR、面试官和业务负责人，避免写成内部工作稿。

1. 页面读者 ​

项目页的第一读者是外部评估者，主要包括：

  • HR：快速判断项目体量、岗位匹配度、关键词和业务价值。
  • 面试官：判断候选人是否真的理解业务、数据、产品和落地过程。
  • 业务负责人：判断候选人是否能把复杂问题拆成可交付的产品系统。

因此，页面要直接回答四个问题：

  • 这是一个什么项目？
  • 项目为什么重要？
  • 我在里面做了什么？
  • 最后形成了什么结果？

2. 总体原则 ​

项目案例页要像正式作品集，不像制作过程记录。

必须做到：

  • 以读者能理解的业务语言表达，不写内部讨论过程。
  • 突出个人能力边界，明确哪些是自己主导，哪些是协作参与。
  • 用数据、模块、客户、成果、视频或截图证明项目真实存在。
  • 重点表达项目价值、业务复杂度、数据复杂度和个人贡献。
  • 可以详细，但要让重点可扫读。

避免出现：

  • “下面先放一组……”
  • “选择逻辑是……”
  • “后续可以补充……”
  • “这个页面建议……”
  • “为了让面试官理解……”
  • “素材规划 / 待确认数据 / 制作说明”
  • “这是我给你的资料 / 项目资料证明……”

这些内容只适合放在内部工作文档，不应出现在正式项目页。

3. 标准页面结构 ​

每个重点项目建议按以下结构组织。

3.1 项目标题 ​

格式：

md
# 项目名称 / 英文名或产品名

标题要清楚，不要过度包装。若项目属于某公司或某产品线，可在正文中说明，不必标题过长。

3.2 项目一句话定位 ​

放在标题下方，用 1 段话概括项目体量。

应包含：

  • 产品类型
  • 使用对象
  • 建设周期或迭代周期
  • 核心能力
  • 个人角色或项目价值

示例表达：

md
一个从 0 到 1 建设、持续迭代 2-3 年的房地产大数据决策系统，覆盖产品框架、数据体系、指标口径、GIS 地图、图表分析、市场查询、客户定制和长期运营维护。

3.3 第一屏项目摘要 ​

第一屏要让 HR 快速抓到重点。建议包含一组关键词或指标。

可包含：

  • 项目周期
  • 产品类型
  • 服务对象
  • 核心模块数
  • 数据链路或核心数据对象
  • 我的角色
  • 关键成果

示例：

md
项目关键词：2-3 年从 0-1 到长期维护、6 大核心模块、房地产全生命周期数据链路、B 端客户交付。

3.4 项目定位 ​

回答“这个产品是什么”。

要写清楚：

  • 面向谁
  • 解决什么问题
  • 为什么这个问题重要
  • 产品核心能力是什么

如果是 B 端系统，要强调业务场景和决策价值，不要写成普通功能介绍。

3.5 项目背景 ​

回答“为什么要做这个项目”。

推荐写法：

  • 先说明行业或业务环境。
  • 再说明原有方式的问题。
  • 最后说明项目要解决的核心矛盾。

可使用表格呈现：

问题具体表现对业务的影响
数据分散多来源、多主体、多口径需要人工拼接，判断效率低
口径不统一同一指标在不同页面或报表中不一致影响客户信任
分析链路长查询、筛选、统计、制图依赖人工难以复用

3.6 项目目标 ​

目标不要只写“提升效率”“优化体验”，要拆成可落地的层级。

推荐三层结构：

  • 数据目标：整合哪些数据，统一哪些对象和口径。
  • 产品目标：形成哪些模块、路径、功能。
  • 业务目标：帮助用户做哪些判断和决策。

3.7 我的角色 ​

这是项目页最重要的部分之一。必须明确个人贡献边界。

建议包含：

  • 我的角色名称
  • 我主导的工作
  • 我参与协同的工作
  • 团队负责的工作

推荐表格：

工作方向我的职责
产品框架设计梳理一级模块、二级功能、用户路径
数据体系梳理梳理数据对象、关系、指标口径
功能原型设计输出模块原型、流程和交互说明
技术方案协同参与前后端方案讨论，说明产品逻辑和实现边界
客户需求对接将客户需求沉淀为可复用产品能力

表达边界要真实：

  • 可以写“我主导产品框架 / 数据关系 / 原型设计”。
  • 如果技术方案由开发主导，应写“我参与技术方案讨论，帮助拆解产品逻辑和实现边界”。
  • 不要把开发实现写成自己完成，除非确实是个人项目。

3.8 我的核心贡献 ​

建议在“我的角色”之后增加一段更直接的贡献归纳。

格式：

贡献类型具体贡献价值
业务抽象将复杂业务拆成模块和路径降低理解成本
数据建模梳理对象、关系、口径支撑长期维护
产品设计将数据能力转成查询、图表、地图、榜单支撑业务使用
客户落地将个性化需求沉淀为可复用功能提升交付能力

这部分要让面试官快速看到“这个项目里最像我的能力是什么”。

3.9 建设过程 ​

建设过程要写关键阶段，不写流水账。

可按以下方式：

  • 竞品研究与方向判断
  • 产品架构梳理
  • 数据体系建设
  • 指标口径与筛选体系
  • 1.0 版本落地
  • 后续迭代与客户适配

每个阶段回答：

  • 当时的问题是什么？
  • 我做了什么判断？
  • 最后沉淀成什么产品能力？

3.10 核心模块设计 ​

适合复杂系统项目。每个模块建议写：

  • 模块服务什么场景
  • 主要功能是什么
  • 数据或产品难点在哪里
  • 对客户有什么价值

模块数量较多时，不要写成菜单清单，要写成业务能力。

示例模块表达：

md
土地市场模块服务开发商拿地、市场监测和企业研究场景。它包括动态监测、宗地查询、企业查询、排行榜和定制土地组。模块难点在于时效性和口径，因为公告、预申请、成交、竞价等状态持续变化，且不同状态对应不同字段。

3.11 数据如何驱动产品 ​

凡是数据产品、AI 产品、知识系统，都建议保留这一节。

应写清：

  • 数据从哪里来
  • 数据对象是什么
  • 数据之间有什么关系
  • 口径如何统一
  • 数据如何变成产品功能
  • 数据如何影响客户决策或用户信任

对于复杂数据项目，可以加入数据链路：

text
数据来源 -> 数据对象 -> 指标口径 -> 分析模块 -> 业务决策

对于 AI 或知识库项目，可以改成：

text
知识来源 -> 结构化规则 -> 案例反馈 -> 结果验证 -> 用户信任

3.12 项目成果 ​

项目成果要尽量数据化。

至少从四个角度写：

  • 产品成果
  • 数据成果
  • 客户 / 业务价值
  • 个人能力沉淀

可量化内容优先：

  • 模块数量
  • 功能数量
  • 视频 / 原型 / 文档数量
  • 核心数据表或数据对象数量
  • 覆盖市场 / 客户 / 场景数量
  • 建设周期和维护周期
  • 客户类型或公开客户背书

如果具体数据不能公开，可以用脱敏表达：

  • 多家头部客户
  • 多类核心数据对象
  • 长期交付和维护
  • 覆盖完整业务链路

3.13 为什么体现我的优势 ​

项目页结尾要收束到个人能力，避免只停在项目介绍。

建议从能力层级总结：

能力层级项目中的体现
行业理解理解业务链路和用户决策场景
数据理解识别数据对象、口径差异和关系
产品抽象把复杂问题拆成模块、路径和交互
业务落地把能力包装成客户可理解、销售可讲、交付可用的材料
长期维护持续调整数据、口径、客户需求和产品边界

最后用一句话总结项目价值和个人价值。

4. 视频和图片展示标准 ​

视频和图片承担项目证据的作用。

4.1 是否需要视频 ​

适合放视频的项目：

  • B 端系统
  • 数据看板
  • 地图 / 图表 / 查询系统
  • AI 原型
  • 交互流程复杂的工具

不适合强行放视频的情况：

  • 视频包含敏感客户信息
  • 视频里有地址栏、个人中心、账号、联系人等信息
  • 视频画面停留在加载态或空状态
  • 视频只能证明“点了某个按钮”，但不能证明项目价值

4.2 视频位置 ​

推荐：

  • 第一屏可放 1 个最能代表项目价值的视频。
  • 核心模块处放对应模块的视频。
  • 视频跟着内容走，不要全部堆在页面顶部。

不推荐：

  • 顶部固定轮播。
  • 一次性堆满所有视频。
  • 一行多个高信息密度视频。
  • 让视频控件、播放速度浮层或进度条干扰正文。

4.3 视频样式 ​

高信息密度系统界面，一行放一个视频。

要求：

  • 使用 MP4 / WebM，不优先使用 GIF。
  • 每段控制在 6-15 秒。
  • 静音循环播放即可。
  • 不展示原生播放器控件，避免播放速度浮层干扰正文。
  • 视频要尽量清晰，能看清地图、表格、图表和关键指标。
  • 视频下面要写明“证明了什么能力”。

视频说明写法：

  • 宏观观察：趋势判断能力。
  • 土地市场：时效监测能力。
  • 新房查询：竞品分析能力。
  • 新房排行：指标比较能力。
  • 二手房查询：存量市场验证能力。

4.4 图片标准 ​

图片应证明：

  • 系统真实存在。
  • 功能模块完整。
  • 业务流程清晰。
  • 数据和指标有深度。

图片不应出现：

  • 未脱敏客户数据。
  • 账号、手机号、联系人、内部地址。
  • 无意义空状态。
  • 裁剪过度、看不出产品内容的截图。

5. 文字表达标准 ​

5.1 要写得像真实经历 ​

推荐表达：

  • “我承担核心数据产品角色……”
  • “我主导产品框架和数据对象梳理……”
  • “我参与前后端技术方案讨论，帮助拆解产品逻辑和实现边界……”
  • “这个项目让我理解到数据资产的重要性……”

避免表达：

  • 过度夸张，如“全面颠覆”“革命性创新”。
  • 过度 AI 味，如连续使用对立式套话。
  • 内部说明，如“根据资料可以看出……”
  • 制作口吻，如“下面先放”“后续补充”“选择逻辑”。

5.2 高亮标准 ​

高亮只标重点，不标整段。

适合高亮：

  • 项目周期
  • 核心模块数量
  • 核心数据链路
  • 我的角色
  • 数据资产、口径、客户交付等关键词
  • 可量化成果

不适合高亮：

  • 普通形容词
  • 长句
  • 内部制作说明

6. 数据成果标准 ​

每个项目尽量补齐以下数据。

类型示例
项目规模周期、模块数、功能数、页面数
数据规模数据对象、指标、表结构、文档、规则数量
业务规模客户类型、服务对象、覆盖场景
交付成果原型、文档、视频、上线版本
个人贡献主导内容、参与内容、协作边界
结果价值效率提升、复用能力、客户交付、决策支持

如果没有精确数字，先用可信的脱敏表达，后续再替换为准确数据。

7. 禁止出现在正式页面的内容 ​

正式项目页不要出现：

  • 内部沟通话术。
  • 对页面怎么制作的说明。
  • 对素材怎么选择的说明。
  • “需要继续补充或确认的数据”。
  • “建议展示的视频片段”。
  • “后续展示素材规划”。
  • “给 HR / 面试官看，所以……”这类显性解释。
  • 未确认、无法公开的客户数据。
  • 没有脱敏的敏感截图和视频。

这些内容可以放在内部备忘、任务文档或本标准文档中。

8. 项目页交付检查清单 ​

每个项目页完成后，按以下清单检查。

  • 是否第一屏能说明项目是什么、给谁用、我做了什么？
  • 是否明确了我的角色和贡献边界？
  • 是否写清楚项目背景、目标、建设过程和成果？
  • 是否体现数据、业务、产品之间的关系？
  • 是否有可扫读的重点高亮？
  • 是否有量化或半量化成果？
  • 是否有视频 / 图片 / 链接等证据？
  • 视频是否清晰、脱敏、无多余控件？
  • 是否删除了制作过程、素材选择、后续补充等内部话术？
  • 页面结尾是否回收到我的能力优势？

9. 不同项目类型的适配 ​

9.1 公司 B 端数据产品 ​

重点：

  • 业务场景
  • 数据链路
  • 指标口径
  • 客户价值
  • 长期维护
  • 跨团队协同

9.2 AI / 个人项目 ​

重点：

  • 问题来源
  • 产品假设
  • 知识或数据结构
  • 原型能力
  • 用户反馈
  • 结果准确性或可验证性

9.3 知识库 / 内容系统 ​

重点：

  • 知识来源
  • 分类体系
  • 结构化方法
  • 检索和复用能力
  • 长期维护机制
  • 对个人工作方式的改变

9.4 考试 / 工具类项目 ​

重点：

  • 使用场景
  • 效率提升
  • 自动化流程
  • 题库或数据结构
  • 反馈闭环
  • 实际使用结果

10. CityMap 项目形成的可复用标准 ​

CityMap 项目的最终展示标准可以总结为：

  • 第一屏先给结论。
  • 项目定位要讲业务价值。
  • 角色边界要真实清晰。
  • 数据链路要作为核心优势。
  • 视频要作为证据，不作为装饰。
  • 视频一行一个，保证系统细节可读。
  • 正式页面不出现制作说明。
  • 结尾回到个人能力和长期沉淀。

后续项目都应沿用这个标准，再根据项目类型调整模块顺序和证据材料。
